# Multi-Catalog / Multi-Server Implementation Analysis

Recap of every component of the multi-server (union catalogue) implementation:
what was chosen, why, the trade-offs, and a regression-risk score where 0 is no
risk and 10 is high risk. Companion to the user-facing guide in
[docs/MULTI_SERVER.md](docs/MULTI_SERVER.md).

## Component recap

| # | Component | What was chosen and why | Pro | Con / trade-off | Risk |
|---|-----------|------------------------|-----|-----------------|:---:|
| 1 | **`music_servers` registry = single source of truth** (table + `/api/servers` CRUD + setup wizard UI) | ALL media-server settings, the default included, live only in the registry. `init_db` migrates a legacy install's `app_config` rows into it once and deletes them; the setup wizard writes to the registry (`save_default_server_settings`); `config` module globals are a read-only projection of the default row, refreshed at import and on registry change, so every legacy read keeps working. `save_config_values` refuses media keys, so the duplication can never come back. | One persistent home, no split-brain possible; add/edit/test/delete servers at runtime; secrets masked with the `__unchanged__` sentinel; env vars remain first-boot seed material only. | Config globals are a per-process cache of the row: a registry change needs `refresh_config()` + worker restart to propagate (both automatic on default-server edits). | **3** |
| 2 | **`track_server_map`** (canonical id <-> per-server provider id, PK `(item_id, server_id)`) | One mapping row per (song, server) instead of duplicating catalogue rows per server. | Union catalogue stays deduplicated; a song analyzed once is playable on every server that has it; FK cascade keeps it consistent through id rewrites. | One provider id per song per server (a server with true duplicates keeps only one); table must be actively maintained (sweep/prune). | **3** |
| 3 | **ContextVar active-server context** (`use_server`, `active_creds/type/libraries`) | Providers read the bound server first and **fall back to the config globals when unset** (which are themselves the projection of the registry's default row, see #1), so an unset context means "the default server" exactly as the historical single-server behavior did. | Zero-regression guarantee for single-server installs; thread/async safe; one mechanism serves HTTP requests, workers, and cron. | Every provider call site had to be threaded through it; a forgotten binding silently targets the default (invisible bug class). | **3** |
| 4 | **Canonical content id = SHA-256(Chromaprint)** (`fp_<hex>`, intact chromaprint kept in `score.chromaprint`) | Identity comes from the audio content itself, not from any server or ML model, so the same song on Jellyfin and Navidrome is *the same row*. | Stable across servers, model upgrades, and re-analysis; exact-equality lookups; enables the whole union design. | Requires downloading every track once for the fpcalc backfill (heavy first run); needs the `fpcalc` binary (guarded: absent = feature off); 120s-window edge cases (e.g. a track vs its extended mix) can collide. | **6** |
| 5 | **Canonicalization relabel** (`fingerprint_canonicalize`: rewrite `item_id` provider->fp across score/playlist/embeddings, merge chromaprint duplicates) | Reuses the battle-tested provider-migration key-rewrite (FK drop/re-add, temp map table, single transaction) instead of new machinery. | Transactional (all-or-nothing); idempotent; duplicate rows merged with mappings preserved. | It is still a destructive whole-catalogue key rewrite; every index must rebuild afterwards; a failure mid-scheme leaves a mixed keyspace (handled by identity fallback, but messy). | **7** |
| 6 | **Union analysis** (`run_analysis_task`: phases run **inline** in one high-queue job, default first; per-phase alignment of remaining servers only) | Inline phases keep the proven one-level topology (parent on `high`, album jobs on `default`) - nesting phases as queue jobs deadlocked with 1 worker per queue (found live, fixed). | Every server's unique songs get analyzed exactly once; shared songs skipped via the map; one index build at the end; cancel/progress on a single task id. | Long single job; a mid-run crash re-runs from album-level dedup (cheap but not free); N servers = N sweep alignments per run. | **5** |
| 7 | **Matching sweep** (`multiserver_sync`: path -> path-tail -> exact-meta -> normalized-meta tiers; full-refresh mode prunes stale mappings + stores `track_count`; fetch bound to the server with its own library filter) | Metadata/path matching aligns ~99% of overlapping libraries with **zero downloads**; Chromaprint is only the fallback identity for what the sweep misses. | Fast (seconds vs hours), incremental, cancellable, live progress; prune keeps counts truthful; never touches the analyzed catalogue itself. | Heuristic: differently-tagged/pathed copies of the same song are missed (then re-analyzed and merged later by chromaprint - wasted model work); never-matching tracks make every sweep re-fetch the full remote catalogue. | **4** |
| 8 | **One shared index per feature + availability mask** (`PagedIvfIndex._availability_mask`, 30s cache; artist index exempt via `track_scoped=False`) | One union index instead of N per-server copies; the mask filters candidates **before** distance ranking. | RAM/build cost independent of server count; results are always playable on the selected server; no result-count starvation from post-filtering. | Touches the hottest code path in the app; mask staleness window (30s TTL); background jobs on the *default* server currently rank union-wide (dropped only at playlist translation). | **5** |
| 9 | **Symmetric id translation boundary** (`canonical_input_ids` in, `translate_ids` out; optional `server` param = name or id, absent = default) | One resolver for every input seed and one choke point (`_to_server_ids`) for every output, both fail-open, so no endpoint invents its own translation. | No v2 API, no new mandatory fields; clients round-trip their own server's ids; fp_ ids can never leak to a provider. | Fail-open means a mapping problem degrades silently (wrong-but-working ids) instead of erroring loudly. | **3** |
| 10 | **IVF build memory rework** (`IVF_TRAIN_SAMPLE_MAX` cap, cells streamed to Postgres, SemGrove disk memmap, in-place normalization) | Needed so ONE union-sized index can build inside the same RAM envelope as before. | Bounded build memory on 500k+ union catalogues; no triple index copies. | Affects **every** install including single-server; k-means sample cap slightly changes cell layout; the old `store_paged_ivf` path is now dead code. | **6** |
| 11 | **Per-server scheduled tasks** (cron `options` JSONB, `server_scope` all/default; clustering computed once then playlists translated per server; sonic/radio run inside each server's context) | Compute once, deliver everywhere: expensive work (clustering) is never repeated per server, only the playlist output is. | Playlists appear on every server with correct native ids; per-server listening history respected for sonic/radio. | A failure on one server's playlist delivery is logged but the playlist set may diverge between servers until the next run. | **2** |
| 12 | **Selector UI + dashboard status** (sidebar dropdown injecting `server=` name; per-server matched/`track_count` bars) | Central fetch-wrapper injects the selection so no page-by-page changes; dashboard measures each server against its own library size captured by the sweep. | One line of integration per page (none); counts finally mean "how much of THIS server is aligned". | `track_count` is only as fresh as the last sweep; dropdown state is per-browser (localStorage). | **1** |

## The three decisions that shape everything

1. **Identity by content, not by server** (#4/#5): the most consequential and
   highest-risk choice - a destructive migration and a heavy first backfill -
   but it is what makes "the same song on three servers" a single row instead
   of three, and it is immune to model changes.
2. **Union catalogue + one masked index** (#6/#8): analysis cost scales with
   *unique* songs, index cost stays constant with server count. The alternative
   (per-server indexes) was rejected for RAM.
3. **Fallback-to-config context** (#3/#9): the whole feature is invisible
   unless a server is bound - that is the mechanism that kept 2,100+ existing
   tests green and single-server installs byte-identical.

## Risk reading

The 6-7 risk rows (#4, #5, #10) share one property: they run rarely (first
analysis after upgrade, index rebuilds) but touch everything when they do. The
steady-state paths - searching, playlists, per-server scoping - sit at 1-3
because they fail open to the historical behavior at every step.
